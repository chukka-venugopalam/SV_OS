## AD-012: Flat Domain Models over Polymorphic Inheritance

| Field             | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Date**          | 2026-06-30                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Context**       | Phase 2.4 implemented 20 SQLAlchemy models. The user's spec listed Subject, Concept, Technology, and Prerequisite as separate entities, but the existing PostgreSQL schema stores all node types in a single `knowledge_nodes` table and all edge types in a single `knowledge_edges` table.                                                                                                                                                                           |
| **Problem**       | Should Subject/Concept/Technology be separate ORM models with polymorphic inheritance, or should they be values of a `NodeType` enum on a flat KnowledgeNode model? Should Prerequisite be a subclass of KnowledgeEdge?                                                                                                                                                                                                                                                |
| **Alternatives**  | 1. **Polymorphic inheritance (single-table)** — Subject, Concept, Technology as subclasses of KnowledgeNode; Prerequisite as subclass of KnowledgeEdge. Requires `polymorphic_on` discriminator column.                                                                                                                                                                                                                                                                |
|                   | 2. **Flat models** — Single KnowledgeNode with `node_type` enum; single KnowledgeEdge with `relationship_type` enum. Subject/Concept/Technology/Prerequisite are query filters, not ORM classes.                                                                                                                                                                                                                                                                       |
|                   | 3. **Separate tables** — Subject, Concept, Technology each get their own table. Requires schema redesign and JOIN queries.                                                                                                                                                                                                                                                                                                                                             |
| **Chosen**        | Option 2 — Flat models with enum discriminators                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Reasoning**     | Polymorphic inheritance (Option 1) required `polymorphic_identity='knowledge_node'` on the base class, but that value is not in the `NodeType` Postgres enum — causing insert/query failures. Fixing this would require adding a fake enum value or making all queries use subclasses. Separate tables (Option 3) would break the existing schema and add unnecessary JOIN complexity. Flat models keep the schema clean, queries simple, and the codebase extensible. |
| **Trade-offs**    | + Clean, simple queries                                                                                                                                                                                                                                                                                                                                                                                                                                                | + No polymorphic ORM complexity | + All node types share one table (as designed) | - Subject/Concept/Technology/Prerequisite are not first-class ORM classes | - Domain model does not directly map 1:1 to the user's entity list | - Type-specific queries require `.where(KnowledgeNode.node_type == 'subject')` instead of `session.query(Subject)` |
| **Future Impact** | If type-specific columns are needed (e.g. `Subject` has a `department` column), a separate table would be needed. The flat model can be extended with JSONB `extra_metadata` without schema changes.                                                                                                                                                                                                                                                                   |

## AD-013: Shared Base Mixin with UUID PK, Timestamps, Soft-Delete, and Version

| Field             | Value                                                                                                                                                                                                                                                                             |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Date**          | 2026-06-30                                                                                                                                                                                                                                                                        |
| **Context**       | Every domain model needs common lifecycle columns.                                                                                                                                                                                                                                |
| **Problem**       | Should every model define these columns individually, or should they inherit from a shared mixin? How should the `metadata` column name conflict with `DeclarativeBase.metadata` be handled?                                                                                      |
| **Alternatives**  | 1. **Inherit from shared `AppBaseMixin`** — all models get UUID PK, timestamps, soft-delete, version. DB column `metadata` exposed as Python `extra_metadata` to avoid conflict.                                                                                                  |
|                   | 2. **Define per-model** — each model defines its own columns. More explicit but highly repetitive.                                                                                                                                                                                |
|                   | 3. **No soft-delete/version** — only UUID PK and timestamps. Less overhead but harder recovery and no optimistic locking.                                                                                                                                                         |
| **Chosen**        | Option 1 — Shared mixin with all lifecycle columns                                                                                                                                                                                                                                |
| **Reasoning**     | A shared mixin ensures consistency, reduces boilerplate, and provides soft-delete (for data recovery) and version (for optimistic locking) on every entity. The `metadata` → `extra_metadata` rename is a one-time cost that avoids the `DeclarativeBase` reserved-name conflict. |
| **Trade-offs**    | + Consistent lifecycle columns across all 20 tables                                                                                                                                                                                                                               | + Soft-delete enables safe recovery | + Version enables optimistic locking later | - `extra_metadata` is a non-obvious Python attribute name | - All tables get `is_deleted` and `version` columns even when not strictly needed (e.g. `activity_logs`, `search_history`) | - Minor storage overhead per table |
| **Future Impact** | Adding columns to the mixin automatically adds them to all 20 tables. Removing soft-delete from specific tables requires refactoring those models to not use the mixin.                                                                                                           |

## AD-014: Alembic Migration Strategy

| Field              | Value                                                                                                                                                                                                                                                                                                                                                                |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Date**           | 2026-07-01                                                                                                                                                                                                                                                                                                                                                           |
| **Context**        | Phase 2.5 implemented the database persistence layer. The existing `database/schema.sql` raw SQL script was being loaded on Docker startup via `docker-entrypoint-initdb.d`. The SQLAlchemy domain models (Phase 2.4) are now the canonical schema definition.                                                                                                       |
| **Problem**        | How should schema changes be managed going forward? Should we continue using raw SQL scripts, switch to Alembic migrations exclusively, or use a hybrid approach?                                                                                                                                                                                                    |
| **Alternatives**   | 1. **Alembic-only** — All schema changes go through Alembic migrations. The raw SQL init script is removed from Docker Compose. Developers run `alembic upgrade head` after pulling changes.                                                                                                                                                                         |
|                    | 2. **Raw SQL only** — Continue using `schema.sql` and manual migration scripts. No Alembic dependency.                                                                                                                                                                                                                                                               |
|                    | 3. **Hybrid** — Alembic for development (autogenerate), raw SQL for production deployment.                                                                                                                                                                                                                                                                           |
| **Chosen**         | Option 1 — Alembic-only                                                                                                                                                                                                                                                                                                                                              |
| **Reasoning**      | Alembic provides deterministic, reversible migrations that are tracked in version control alongside the models. Autogenerate detects model changes and produces migration candidates. Raw SQL scripts are error-prone (no rollback, no dependency tracking, no autogenerate). The Docker init script has been removed and replaced with a post-start migration step. |
| **Trade-offs**     | + Reversible migrations                                                                                                                                                                                                                                                                                                                                              | + Autogenerate support                                                                  | + Version-controlled alongside models          | + Consistent naming convention enforced by Base.metadata | - Requires developers to run `alembic upgrade head` after pulling | - Autogenerated migrations must be reviewed manually | - Additional complexity compared to raw SQL |
| **Implementation** | - Migration 0001: Create extensions (uuid-ossp, pgcrypto, pg_trgm, unaccent, btree_gin, btree_gist)                                                                                                                                                                                                                                                                  | - Migration 0002: Create all 20 tables, 13 enums, indexes, constraints, triggers, views | - Extensions are documented with justification | - Every index is documented with rationale               | - All migrations are fully reversible                             |
| **Future Impact**  | Adding columns to a model requires: (1) Edit model, (2) Run `alembic revision --autogenerate -m "description"`, (3) Review and edit the generated migration, (4) Run `alembic upgrade head`, (5) Commit both the model change and the migration.                                                                                                                     |

## AD-015: Repository Pattern with Unit of Work

| Field             | Value                                                                                                                                                                                                                                                                                                                                                                                |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Date**          | 2026-07-04                                                                                                                                                                                                                                                                                                                                                                           |
| **Context**       | Phase 2.7 implements the data-access layer. Phase 2.4 created 20 SQLAlchemy models, Phase 2.5 created the database persistence (migrations), and Phase 2.6 created 127 Pydantic DTOs. The missing link is the repository layer that abstracts SQLAlchemy from the rest of the application.                                                                                           |
| **Problem**       | How should database access be structured to prevent SQLAlchemy from leaking into business services and API routes? How should transactions be managed across multiple repository operations?                                                                                                                                                                                         |
| **Alternatives**  | 1. **Repository pattern + Unit of Work** — Generic `BaseRepository[T]` with CRUD operations, feature-specific repositories for domain queries, `UnitOfWork` for transaction management.                                                                                                                                                                                              |
|                   | 2. **Active Record** — Models manage their own persistence (e.g., `user.save()`, `user.delete()`). Simple but couples business logic to persistence.                                                                                                                                                                                                                                 |
|                   | 3. **DAO pattern** — Data Access Objects with minimal abstraction. Thin wrapper over SQLAlchemy without generics or UoW.                                                                                                                                                                                                                                                             |
|                   | 4. **CQRS with separate read/write models** — Separate interfaces for reads and writes. Overengineered for current scope.                                                                                                                                                                                                                                                            |
| **Chosen**        | Option 1 — Repository pattern with Unit of Work                                                                                                                                                                                                                                                                                                                                      |
| **Reasoning**     | The repository pattern provides a clean abstraction boundary between persistence and business logic. The Unit of Work ensures transaction consistency across multiple repository operations. Generics reduce boilerplate while feature-specific repositories provide domain-friendly query methods. The 'no SQLAlchemy leakage' rule is enforced by the exception translation layer. |
| **Trade-offs**    | + Clean separation of concerns (persistence vs. business logic)                                                                                                                                                                                                                                                                                                                      | + Transaction consistency via UoW                                                                              | + Soft-delete applied automatically                                          | + Generic CRUD reduces boilerplate by ~70%                     | + QueryBuilder provides composable, type-safe query construction                                                            | - More files than Active Record (~20 vs. 1)                                               | - Feature repos duplicate some query patterns (acceptable for clarity) | - UoW adds complexity for simple read-only operations (mitigated by allowing direct repo injection) |
| **Rules**         | 1. Repositories must **never** call `session.commit()` — only `session.flush()`                                                                                                                                                                                                                                                                                                      | 2. Repositories must **never** expose SQLAlchemy exceptions — always translate to `RepositoryError` subclasses | 3. All queries filter soft-deleted records by default (`is_deleted = False`) | 4. The `UnitOfWork` is the single point of transaction control | 5. Graph operations that are **data access only** go in `GraphRepository`; graph algorithms go in a separate `GraphService` | 6. Business logic belongs in the Application Service layer (Phase 3), not in repositories |
| **Future Impact** | Adding a new entity requires: (1) Create feature repository extending `BaseRepository[T]`, (2) Add repository accessor property to `UnitOfWork`, (3) Add dependency injector in `deps.py`, (4) Export from `repositories/__init__.py`.                                                                                                                                               |
