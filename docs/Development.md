# Development Guide

## Architecture Overview

SV-OS follows Clean Architecture with three layers:

```
Routes (api/) → Services (services/) → Repositories (repositories/) → Database
```

- Routes handle HTTP request/response
- Services contain business logic
- Repositories handle data access

## State Management (Frontend)

Three tools with distinct responsibilities:

| Concern      | Tool                  | Purpose                                          |
| ------------ | --------------------- | ------------------------------------------------ |
| Server State | TanStack React Query  | API data caching, refetching, optimistic updates |
| Client State | Zustand               | UI preferences, graph viewport, active filters   |
| Form State   | React Hook Form + Zod | Typed form validation                            |

## Graph Architecture

The knowledge graph uses a **relational adjacency list** pattern (not Neo4j).

- Graph traversal via PostgreSQL recursive CTEs
- Depth limited to 10 levels
- Edge directions: forward, bidirectional, unidirectional
- All mutations go through the Service layer

## API Conventions

- Versioned endpoints: `/api/v1/`
- Standard response: `{success, message, data, errors, timestamp, request_id}`
- Pagination: `{items, total, page, page_size, total_pages}`
- Authentication: Bearer JWT tokens via Supabase Auth

## Backend Development

### Prerequisites

- Python 3.12+
- PostgreSQL 16 (via Docker Compose: `docker compose up -d`)

### Setup

```bash
cd apps/api
python -m venv .venv                 # Create virtual environment
.venv\Scripts\activate               # Activate (Windows)
source .venv/bin/activate            # Activate (macOS/Linux)
pip install -e .                     # Install dependencies
pip install -e ".[dev]"              # Install dev dependencies (optional)
```

### Running

```bash
cd apps/api
.venv\Scripts\python -m uvicorn app.main:app --reload --port 8000
```

The API will be available at `http://localhost:8000` with Swagger docs at `http://localhost:8000/docs`.

### Testing

```bash
cd apps/api
.venv\Scripts\python -m pytest tests/ -v
```

### Database Migrations

Schema changes are managed through **Alembic** migrations in `apps/api/alembic/versions/`.

#### Running Migrations

```bash
cd apps/api
.venv/Scripts/python -m alembic upgrade head
```

#### Creating a New Migration

```bash
cd apps/api
.venv/Scripts/python -m alembic revision --autogenerate -m "description"
```

Always review autogenerated migrations before committing.

#### Rolling Back

```bash
cd apps/api
.venv/Scripts/python -m alembic downgrade -1    # One step back
.venv/Scripts/python -m alembic downgrade base    # All the way back
```

#### Loading Seed Data

```bash
cd /path/to/project/root
./database/scripts/seed.sh
```

#### Database Utilities

| Script                              | Purpose                               |
| ----------------------------------- | ------------------------------------- |
| `database/scripts/reset.sh`         | Drop, recreate, migrate, and seed     |
| `database/scripts/seed.sh`          | Load seed data in dependency order    |
| `database/scripts/backup.sh`        | Custom-format PostgreSQL backup       |
| `database/scripts/restore.sh`       | Restore from backup with confirmation |

#### Migration Tests

```bash
cd apps/api
.venv/Scripts/python -m pytest tests/migrations/ -v
```

### Middleware Stack

Middleware is registered in `app/main.py` in this order:

1. **CORSMiddleware** — Handle preflight requests
2. **GZipMiddleware** — Compress responses >1KB
3. **TrustedHosts** — Validate Host header
4. **SecurityHeaders** — Set secure HTTP headers
5. **RequestID** — Assign unique ID to each request
6. **CorrelationID** — Propagate trace-wide ID
7. **RequestTiming** — Measure request duration
8. **RateLimit** — Stub (no enforcement yet)

### Adding a New Endpoint

1. Create a route module in `app/api/v1/endpoints/` (e.g. `nodes.py`)
2. Define a router with the v1 prefix
3. Add the endpoint function with Pydantic request schemas
4. Inject dependencies via `Depends()` (DB session, settings, context)
5. Register the router in `app/api/v1/router.py`
6. Write tests in `tests/`

### Exception Handling

```python
from app.exceptions import NotFoundError

# Raise in service layer:
raise NotFoundError(message=f"Node '{slug}' not found")

# Caught by global handler → consistent error response
```

Available exceptions: `NotFoundError` (404), `ConflictError` (409), `ValidationError` (422), `UnauthorizedError` (401), `ForbiddenError` (403), `RateLimitedError` (429), `InternalError` (500), `ServiceUnavailableError` (503).

### Logging

```python
from structlog.stdlib import get_logger
logger = get_logger(__name__)

logger.info("node_fetched", node_id=str(node_id), duration_ms="45.23")
```

Logs include `request_id` and `correlation_id` automatically via middleware. Development output is coloured console; production is JSON.
